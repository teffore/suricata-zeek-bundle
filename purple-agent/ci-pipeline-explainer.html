Suricata+Zeek Detection CI

Ephemeral AWS lab. Real network sensor. Coverage-gated every run. ~$0.04 / 8 min / 38 scenarios.

TL;DR. On every manual workflow_dispatch, GitHub Actions spins up a disposable 3-box AWS lab, runs ~220 attacks against it, checks what the network sensor detected, and destroys the lab. Pass/fail is gated on a coverage-% of expected detections, not just alert count.

Topology

All three boxes are ephemeral t3.medium Ubuntu 22.04 instances. The sensor is passive — a mirrored observer, not inline.

┌─────────┐ ┌─────────┐ ┌─────────┐ │ attacker │ ─── intra-SG ──── any ────────────────▶ │ victim │ │ │ ┌─────────┐ │ │ │ nmap │ │ sensor │ │ nginx │ │ hydra │ VPC Traffic│ │ │ sshd │ │ nikto │ Mirror ───▶│ Suricata │◄─────────── │ samba │ │ impacket │ (VXLAN │ + Zeek │ VNI=1 │ Vulhub │ │ nuclei │ to ENI) │ │ victim ENI │ (Log4j + │ │ flightsim│ └─────────┘ │ Spring) │ └─────────┘ └─────────┘ VNI=2 attacker ENI (filter rejects attacker↔victim so we don't double-capture)

Victim traffic is mirrored to the sensor's ENI via AWS VPC Traffic Mirroring. The sensor's NIC is set promiscuous + MTU 9001 to handle the VXLAN encapsulation. Suricata runs on af-packet; Zeek runs alongside with the FoxIO JA4, MITRE BZAR, and Corelight long-connections zkg packages.

What one CI run does

Defined in .github/workflows/validate-detections.yml:

  1. OIDC auth to AWS — no long-lived keys. Assumes an IAM role scoped to EC2 + VPC Traffic Mirror + ENI operations.
  2. Generate an ephemeral ed25519 SSH key, import it to EC2 under a per-run tag (gha-det-<run-id>-<attempt>).
  3. Create a security group — SSH from the GitHub runner's IP only; intra-SG any-to-any so attacker↔victim and VXLAN 4789 flow freely.
  4. Launch 3 EC2 instances in parallel — sensor, victim, attacker. Wait for running state, then for SSH + cloud-init readiness (parallel, not serial — that was a big speedup PR).
  5. Disable source/dest-check on the sensor ENI so mirrored frames with arbitrary inner IPs aren't dropped by the Nitro NIC, then set promisc + MTU 9001.
  6. Install stacks in parallel:
    • sensor — standalone.sh (Suricata 8 + Zeek 8 + custom rules + zkg packages + abuse.ch intel feeds)
    • victim — testing/victim_setup.sh (nginx, sshd, samba, Vulhub Log4Shell on :8983, Vulhub Spring4Shell on :8080)
    • attacker — testing/attacker_setup.sh (nmap, hydra, nikto, impacket, nuclei, flightsim, curl, hping3)
  7. Create 2 traffic-mirror sessions. VNI=1 captures all victim ENI traffic. VNI=2 captures attacker ENI but with a filter that rejects attacker↔victim so session-1 keeps exclusive ownership of intra-lab traffic. This is why the attacker's external egress (DNS tunnel probes, SaaS SNI exfil, flightsim C2 callbacks) gets observed — the victim-side mirror alone couldn't see it.
  8. Smoke-check that VXLAN frames are actually reaching the sensor (catches mirror misconfiguration early).
  9. Run the attack battery — SSH to attacker and invoke sudo timeout 900 bash /tmp/run_attacks.sh <victim_private_ip>. Wall-clock ~3 minutes.
  10. Run verify_alerts.sh on the sensor — the coverage gate. Exits 0 if both gates pass.
  11. Upload artifacts — fast.log, eve.json, zeek logs, coverage JSON.
  12. Teardown (always, even on failure) — mirror sessions → filters → target → terminate instances → delete SG → delete key pair, exact reverse of creation order.

The attack battery

testing/run_attacks.sh — ~220 probes across ~40 categories.

12 parallel background groups

For long-running scans: nmap (SYN / version / vuln / NSE), hydra SSH brute, nikto, impacket lateral SMB, impacket AD chain (GetNPUsers / GetUserSPNs / secretsdump / lookupsid), nuclei CVE sweep, flightsim (AlphaSOC C2/DGA/DNS-tunnel/miner), DoH tunneling (Cloudflare / Google / Quad9), 2024 CVE sigbank (TeamCity + ScreenConnect), HTTP/2 h2c + smuggling, SaaS SNI exfil (Telegram / Dropbox / webhook.site / pipedream / ngrok / trycloudflare / pythonhosted), cryptominer stratum + pool SNI.

Serial inline probes

Fast stuff: web shells, SQLi/XSS/XXE/SSTI/LFI, cloud IMDS SSRF, supply-chain patterns, C2 framework URI replay, reverse-shell patterns, tunneling SNIs.

The gate

testing/verify_alerts.sh joins testing/probe_catalog.json (38 scenarios, each mapping probes → expected Suricata SIDs / Zeek notices / MITRE technique) against what actually fired in eve.json + notice.log. Two gates:

  • Sanity — ≥ 3 alerts total (means the pipeline is alive)
  • Coverage — ≥ 50% of scenarios had any expected SID or notice fire
Current state: 95% (36/38) scenarios PASS. The 2 MISSes are architectural: Zeek 8 can't observe SSH auth events inside the encrypted post-KEX channel, and the abuse.ch intel feed rotates hourly so intersection with our probes is probabilistic — not rule bugs.

Customization

Adding a new probe is a three-file edit:

  1. Add the install line to attacker_setup.sh.
  2. Add the invocation to run_attacks.sh.
  3. Add a catalog entry to probe_catalog.json with expected_suricata_sids, expected_zeek_notices, and a MITRE technique.

Every merged PR refreshes the wiki Detection-Report and Attack-Battery pages. The coverage journey is documented PR-by-PR — from 57% → 95% across eight merged changes.

Why it's useful as a demo

  • Real network-sensor data plane. Suricata+Zeek on mirrored VPC traffic, not a logs-only fixture.
  • Coverage gate, not count gate. A probe that stops firing is immediately visible; a rule change that trips a neighbor is caught.
  • Full teardown contract. Every resource is tagged with the run ID and cleaned up in a guaranteed if: always() finally block — no forgotten AWS bills.
